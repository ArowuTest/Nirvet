# Nirvet recovery manifest

Status: implementation baseline for `GATE_FULL_STACK_RECOVERY_VALIDATION.md`. This manifest is deliberately explicit about what is already recoverable, what is reconstructible, and what still requires validation wiring. A component may not be omitted merely because CI does not currently emulate it.

| Stateful component | Backup source | Restore path | Required recovery validation | Current implementation status |
|---|---|---|---|---|
| PostgreSQL | Consistent `pg_dump`/PITR including migration state and all tenant/platform tables | Existing B8 restore into a fresh database, then migrations/schema-at-head verification | Recorded table counts/checksums, FK/integrity checks, full schema check, FORCE-RLS + owner_bypass verification, non-owner runtime-role assertion, two-tenant non-contamination proof | B8 round-trip exists; full-stack assertions are being added in this item |
| Crypto material / KEK providers | Provider-specific procedure: local master key held separately; Vault/HSM keys backed up through provider procedure; database stores ciphertext/provider metadata, never raw KEK | Re-provision provider and credentials before app startup; use existing provider-tagged/transition-cipher paths where rotation occurred | Provider reachability, decrypt probes for every encrypted domain, per-tenant unwrap, wrong/missing-key negative control, no plaintext fallback | B8 proves one real encrypted sample; domain-wide probe inventory and validation remain in this item |
| Object/blob storage | Provider snapshot/versioned backup for evidence, attachments, exports and air-gap artifacts; metadata references remain in PostgreSQL | Restore bucket/container before serving, preserving object keys, versions and integrity metadata | Manifest/key reconciliation against PostgreSQL references, existence and checksum probes, no orphaned/missing required object | Storage adapters exist; cross-store recovery validator must report unavailable provider as failure, not skip |
| Queue, outbox and idempotency state | Durable PostgreSQL-backed queues/outboxes/idempotency records are included in the database backup; broker-only transient delivery state is reconstructed from durable records | Restore durable records; reconnect brokers/connectors only after certification | No replay of completed SOAR runs, sent notifications or consumed idempotency keys; pending work resumes without auto-firing destructive actions on reconnect | Durable stores exist; replay/staleness assertions are required in this item |
| Secrets and deployment config | Secret manager / operator-controlled deployment backup, outside application data backup | Re-provision all required DSNs, signing material, KMS credentials, JWT/session secrets and policy configuration before boot | Existing deploy-security posture plus explicit required-secret checks; missing secret must abort restored startup with a clear error and no weaker fallback | Existing startup/security checks are reused; restored-mode certification gate is being added |
| Offline content packages | PostgreSQL `content_packages`, `content_artifacts` and `content_lifecycle_audit`, plus original signed bundle/object-store copy where retained | Restore package provenance and lifecycle state; never infer active state from artifact presence alone | Active/quarantined state consistency, signature/provenance availability, no rolled-back/superseded pack reactivation, tenant/global scope intact | Tables landed in `b839cd8`; persistence/live wiring remains a separately reviewed follow-on noted by the item-4 reviewer |
| Audit records | PostgreSQL append-only audit tables and any external audit export/sink | Restore with database and reconnect external sink after certification | Continuity across backup/recovery seam, append-only controls intact, recovery event records actor/backup/restore ids and result | Existing audit controls must be re-run against restored state; seam assertion is part of this item |
| Session and token revocation state | PostgreSQL session-generation/revocation records plus required signing secrets | Restore records, then advance recovery generation before serving | Pre-failure refresh/access tokens cannot silently revive; owner/superuser startup remains rejected | Existing session-generation mechanism is reused; recovery advancement assertion remains to be wired |
| Retention, legal hold and erasure state | PostgreSQL policy/control records plus backup catalogue metadata recording backup time and erasure watermark | Restore snapshot only into maintenance mode, reconcile against authoritative post-backup erasure/hold ledger before certification | Deleted/purged data is not reintroduced into service; legal-hold state and retention floors remain intact | Validator must fail certification when authoritative post-backup tombstone/watermark data is unavailable |
| ClickHouse / analytical event state | ClickHouse-native backup/snapshot or documented reconstructible ingest source, according to deployment profile | Restore ClickHouse backup or replay from the authoritative retained source before certification | Partition/table completeness, tenant labels, event-count/time-window reconciliation and query smoke test | Not exercised by the original PostgreSQL-only B8 drill; deployment-specific validator is mandatory for production evidence |

## Certification rule

A restored deployment is untrusted until every applicable row above has produced positive evidence. “Not configured in this deployment” must itself be proven from the deployment profile; “not checked” is a certification failure. The binary certification model lives in `backend/internal/platform/recovery` and will be wired into restored-mode startup before this PR is ready.
